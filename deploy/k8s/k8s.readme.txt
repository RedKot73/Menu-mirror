1. Первоисточник (Где рождаются данные)
Все начинается с переменных. В зависимости от среды, источник разный:

Локально: Файл .env в корне твоего проекта.

В облаке (Dev/Prod): Переменные CI/CD в настройках GitLab (Variables), которые на лету записываются в файлы dev.env или prod.env.

2. Слой Kustomize (Преобразование данных)
Когда запускается деплой, Kustomize считывает эти .env файлы и создает два типа объектов в Kubernetes:

Secret (s5-db-credentials): Сюда попадает пароль (DB_PASSWORD). В секрете данные хранятся в кодировке Base64.

ConfigMap (s5-db-params): Сюда попадают открытые данные: DB_NAME, DB_USER, DB_PORT.

3. Распределение данных по компонентам
После создания ConfigMap и Secret, данные текут в трех направлениях:

А. В Базу Данных (CloudNativePG)
Здесь работает магия файла db-replacements.yaml:

Kustomize берет текстовое значение DB_NAME из ConfigMap.

«Вклеивает» его в манифест Cluster вместо плейсхолдера change-me.

Оператор базы видит это имя и создает базу данных именно с таким названием.

Б. В Job инициализации (s5-db-init-job)
Контейнер-импортер получает данные через переменные окружения (env):

PGDATABASE и PGUSER подтягиваются напрямую из ConfigMap.

PGPASSWORD подтягивается из Secret.

Благодаря этому скрипт внутри Job точно знает, в какую базу заливать дамп.

В. В Приложение (s5-server)
Твой .NET сервис получает настройки через маппинг в k8s-deployment.yaml:

Переменные типа PrimaryConnection__DB_Name связываются с ключами из ConfigMap.

Пароль связывается с ключом password из Secret.

При старте .NET автоматически собирает эти переменные в строку подключения (Connection String).

4. Поток обновления Образов (CI/CD Flow)
Отдельно текут данные об образах:

GitLab CI собирает образ и вешает на него уникальный тег (например, main-88f21a5b).

Команда kustomize edit set image находит в твоем оверлее секцию images.

Она заменяет старый тег на новый main-88f21a5b.

Kubernetes видит, что в деплойменте изменилась строка образа, и дает команду «перезапустить поды», чтобы скачать новую версию кода.
