# Анализ логики авторизации 2FA и навигации

Этот документ содержит описание текущей реализации механизмов двухфакторной аутентификации (2FA) и логики навигации в приложении.

**Дата проведения**: 2026-04-02 10:28 (UTC)

---

## 1. Действующая логика 2FA-авторизации

Логика разделена на два независимых этапа:

### Этап 1: Первичный вход (Login)
*   **Backend**: При успешной проверке учетных данных, если у пользователя включена двухфакторная аутентификация, сервер возвращает статус `requiresTwoFactor: true` и генерирует **промежуточный JWT-токен** (interim token). Этот токен содержит специальное утверждение (claim) `requiresTwoFactor: true`, которое ограничивает возможности API только методами верификации.
*   **Frontend**: Компонент входа (`LoginPage`) получает данный ответ и выполняет навигацию на страницу `/welcome`. Промежуточный токен сохраняется в локальном хранилище браузера (`localStorage`).

### Этап 2: Верификация (Welcome Page)
*   **Backend**: Метод `VerifyTwoFactor` (GraphQL) извлекает идентификатор пользователя из промежуточного токена и проводит валидацию TOTP-кода.
*   **Режим `TWO_FACTOR_MODE=soft` (Мягкий режим)**: 
    *   При вводе **неверного кода** сервер делает искусственную паузу в **3 секунды** (`Task.Delay(3000)`).
    *   После завершения паузы сервер игнорирует факт невалидности кода и генерирует полноценный токен доступа (Full Access Token), обеспечивая пользователю доступ в систему.
*   **Frontend**: При получении успешного ответа (полноценного токена), фронтенд-компонент выполняет принудительную перезагрузку браузера через `window.location.href = '/DocumentDataSet'`. Это гарантирует выход из SPA-состояния страницы приветствия и полную инициализацию рабочего интерфейса.

## 2. Логика навигации: Текущая реализация

Текущая схема переходов в коде выстроена следующим образом:

### Основной поток при входе
1.  **Форма входа**: Пользователь вводит логин и пароль.
    *   Если 2FA **не требуется**: Происходит полная перезагрузка страницы (`Hard Reload`) на `/DocumentDataSet`.
    *   Если 2FA **требуется**: Происходит переход внутри Angular-приложения (SPA) на страницу `/welcome`.
2.  **Страница верификации (`/welcome`)**: 
    *   Пользователь вводит 6-значный код.
    *   После нажатия кнопки верификации (`Confirm`) и успешного ответа сервера, инициируется принудительное обновление страницы браузера с переходом к целевому разделу `/DocumentDataSet`.

### Технические особенности реализации
*   **Hard Reload**: Использование `window.location.href` вместо `router.navigate` является осознанным архитектурным решением для обеспечения полной очистки памяти и всех временных переменных (signals, states), связанных с процессом 2FA, перед переходом в основную рабочую область.
*   **Промежуточное состояние**: До момента успешной верификации (или срабатывания `soft` режима по запросу), приложение находится в состоянии `pendingTwoFactor`, ограничивая доступ к основному функционалу через JWT-токен с ограниченными правами.
