# Отчет по аудиту зависимостей User & Soldier

Этот отчет содержит результаты анализа связей между сущностями `User` (TVezhaUser) и `Soldier` в бэкенд и фронтенд частях приложения. 

**Дата проведения**: 2026-04-02 10:17 (UTC)

---

## 1. Backend: Анализ (EF Core & GraphQL)

На текущий момент связь между пользователем и солдатом глубоко интегрирована в модели данных и логику авторизации.

### Обнаруженные зависимости

*   **Навигационное свойство**: В файле `TVezhaUser.cs` (строка 278) сохранено поле `public virtual Soldier? Soldier { get; set; }` с атрибутом `[ForeignKey("Id")]`. Это указывает на жесткую связь 1-к-1 на уровне базы данных.
*   **DTО и Маппинг**: `UserInfoDto` включает в себя объект `Soldier? Soldier = null` (строка 265). Расширение `ToInfoDto` (строка 366) автоматически заполняет это поле из навигационного свойства пользователя.
*   **Принудительная загрузка (Eager Loading)**: В GraphQL-слое (`AuthMutation.cs`) в методах `Login` (строки 80–84) и `VerifyTwoFactor` (строки 160–164) прописан явный вызов `LoadAsync()` для подзагрузки данных солдата и его звания. 
*   **REST Контроллеры**: `AccountController.cs` (метод `/me`) возвращает `UserInfoDto`, однако в его основном запросе `UsersQuery()` отсутствует `.Include(u => u.Soldier)`, из-за чего в REST-ответах поле `soldier` обычно равно `null`.
*   **Контекст БД**: В `MainDbContext.cs` зарегистрирован `DbSet<Soldier> Soldiers`. Явных конфигураций Fluent API (OnModelCreating) не обнаружено — связь работает на уровне атрибутов моделей.

## 2. Frontend: Анализ (Angular & GraphQL)

Фронтенд частично зависит от объекта `soldier` для формирования отображаемого имени пользователя.

### Обнаруженные зависимости

*   **Модели данных**: В `auth.models.ts` интерфейс `AuthUser` содержит поля `soldierId: string` и `soldier?: SoldierDto`.
*   **Мутации авторизации**: В `auth.service.ts` запросы `login` и `verifyTwoFactor` в текущий момент **не запрашивают** поле `soldier` в GraphQL-схеме, хотя сервер готов его отдать. 
*   **Логика отображения (Display Name)**: В `auth.service.ts` вычисляемое свойство `displayName` (Signal) пытается построить ФИО пользователя из `u.soldier`. Если объекта солдат нет — возвращается основной `userName`. 
*   **UI Компоненты**:
    *   `navigator.component.ts`: Использует `auth.displayName()` для показа имени в шапке.
    *   `Users.page.html`: Содержит ссылки на `user?.soldier`, которые перестанут работать при удалении объекта.

## 3. Резюме и Рекомендации

| Область | Уровень зависимости | Статус |
| :--- | :--- | :--- |
| **Backend** | **Высокий** | Остаются навигационные свойства и Eager Loading в GraphQL. |
| **Frontend** | **Средний** | Используется для отображения ФИО через Signal-свойство. |

### Рекомендуемые шаги по устранению зависимости

1.  **Backend Cleanup**: Удалить навигационное свойство `Soldier` из `TVezhaUser`, очистить `UserInfoDto` и убрать блоки принудительной загрузки в `AuthMutation.cs`.
2.  **User Identity**: Добавить поле `DisplayName` (или `FullName`) напрямую в таблицу `users` (модель `TVezhaUser`). Это позволит фронтенду получать имя одним полем без доступа к таблице солдат.
3.  **Frontend Cleanup**: Обновить интерфейсы в `auth.models.ts` и упростить логику `displayName` в `auth.service.ts`.
