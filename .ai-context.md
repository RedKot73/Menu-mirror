# Project Ground Truth

This document serves as the absolute source of truth for AI agents interacting with this project. It outlines the system architecture, technology stack, directory structure, navigation mapping, security rules, and infrastructure.

## Architecture Guidelines
The system is built as a highly coupled monolith, where the .NET Backend simultaneously serves a REST API, acts as a GraphQL Server, and hosts the compiled Angular Frontend.

- **"Thin" Backend Architecture**: Avoid complex architectural patterns like `CQRS (MediatR)` or `Repository Pattern`. Focus heavily on using `Entity Framework Core` directly in conjunction with `HotChocolate` via attributes (`[UseProjection]`, `[UseFiltering]`, `[UseSorting]`) to stream data effortlessly.
- **Business Logic Layer**: Complex queries or operations reside in `Services/` and should be implemented as **Extension Methods** layered on top of `DbSet<T>`.
- **Frontend SPA**: Powered by Angular 21.0.5. Source code lives in `/S5Server/Front/src/`. Upon building (via MSBuild Targets in the `.csproj`), the Angular app compiles straight into the backend's `wwwroot` directory to be statically served. **DO NOT** use Razor (`.cshtml`) or Blazor (`.razor`).
- **UI Framework Rules**: **Angular Material ONLY**. The use of Tailwind CSS, Bootstrap, or any custom utility-first framework is strictly forbidden unless explicitly mandated by the user. Rely on native Angular Material properties and components.
- **Database**: Relational data relies entirely on **PostgreSQL** configured via `Npgsql.EntityFrameworkCore.PostgreSQL`, with Code-First schema management via EF Core Migrations (`Migrations/`).

## Naming Conventions
To maintain uniformity across the monorepo, strictly adhere to the following rules:
- **C# (Backend)**: Use `PascalCase` for namespaces, classes, records, properties, methods, and interfaces (with `I` prefix). Use `camelCase` for variables and method parameters.
- **TypeScript/Angular (Frontend)**: Use `camelCase` for variable names, component properties, and functions. Use `PascalCase` for Class and Interface names. Strictly adhere to Angular file naming schemes (`kebab-case.component.ts`, `kebab-case.service.ts`).
- **GraphQL API**: Use `camelCase` for all Queries, Mutations, and property accessors, standardizing JSON responses.

## Security Constraints
- **CORS Policy**: Configured using `AllowedOrigins` from `appsettings.json`. Wildcard origin requests are blocked on production, allowing exceptions only for localized development servers (`localhost:4200`).
- **Authorization Schema**: Access uses JWT Bearer Tokens validated by `HotChocolate.AspNetCore.Authorization` middleware. APIs enforce HTTP `401/403` status outputs in lieu of redirecting automatically.
- **Secrets Management**: Local environment depends entirely on `UserSecretsId`. Staging/Production clusters mount secrets dynamically utilizing Kubernetes mapped Environmental Variables (`.env`) or Azure Identity modules.

## Auth Mechanism (2FA Step-up Flow)

### 2-Step Login Sequence (GraphQL Mutation)
1. **Step 1: Credentials**: `Mutation.Login(userName, password)` returns `AuthPayload`.
   - If User has 2FA enabled: Returns `{ requiresTwoFactor: true, userId, interimToken }`. User is NOT yet fully logged in.
   - If User has NO 2FA: Returns `{ requiresTwoFactor: false, token, user }`. User is fully logged in.
2. **Step 2: Verification**: `AuthService` detects `requiresTwoFactor` and navigates to `/welcome`.
3. **Step 3: TOTP Code**: User enters 6-digit code. `Mutation.VerifyTwoFactor(code)` is called.
4. **Step 4: Completion**: If code is valid (via `OtpNet` verification against secret), returns final `AuthPayload` with JWT and full `User` object. `AuthService` updates state and navigates to `/units`.

### Route Guard Logic
| Guard | Condition | Result |
|-------|-----------|--------|
| `authGuard` | `requiresTwoFactor()` | → `/welcome` |
| `authGuard` | `isAuthenticated()` | Allow access |
| `authGuard` | Neither | → `/login` |
| `twoFactorGuard` | `requiresTwoFactor()` | Allow access to `/welcome` |
| `twoFactorGuard` | Not pending | → `/login` |

### Key Files
- `src/app/auth/auth.service.ts` — State signals: `user`, `pendingTwoFactor`, `requiresTwoFactor`, `isAuthenticated`.
- `src/app/auth/auth.guard.ts` — Global `authGuard` and route-specific `twoFactorGuard`.
- `src/app/pages/welcome/` — `WelcomeComponent` for TOTP entry.
- Backend: `S5Server/GraphQL/Mutation.cs` — `login` and `verifyTwoFactor` methods.
- Backend: `S5Server/Controllers/AccountController.cs` — Legacy REST endpoints (transitioning to GraphQL).

### Development Mode (Soft Mode)
- **Environment Variable**: `TWO_FACTOR_MODE=soft` in `.env`.
- **Behavior**: If 2FA check fails, a warning is logged: `[DEBUG] 2FA Check Failed for user {name}, but access granted due to Debug Mode.`. Access is granted regardless of code accuracy. **DO NOT USE IN PRODUCTION**.

## Infrastructure & DevOps
- **Kubernetes (K8s) & RDBMS**: PostgreSQL runs inside K8s, governed by **CloudNativePG (CNPG)**.
- **Containerization**: Database initialization uses `Dockerfile.kuber`.
- **CI/CD**: GitHub Actions automates builds (`.github/workflows/build-release.yml`), compiling self-contained binaries.
- **Telemetry & Logging**: Uses **Serilog** for logging and Azure Application Insights for production monitoring. Logs are stored in `Logs/s5server-.txt`.
- **Environment Configuration**: K8s bridges variables to `.NET` configuration (e.g., `PrimaryConnection__DB_Host`).

## Runtime & Debugging Rules

This section strictly regulates the application lifecycle within the sandbox: **Setup -> Run -> Cleanup**.

### 1. Environment Setup
Before launching the application, ensure the following environment variables are set to guarantee proper file watching and detailed error reporting:
```bash
export DOTNET_USE_POLLING_FILE_WATCHER=1
export ASPNETCORE_ENVIRONMENT=Development
```

### 2. User Secrets Configuration
Initialize the necessary security credentials and local settings using the following commands:
```bash
dotnet user-secrets set "Authentication:JwtSecret" "<32-char-string>"
dotnet user-secrets set "TOTP:Issuer" "S5Server"
```
Secrets are stored in `~/.microsoft/usersecrets/`.

### 3. Launch Command
Use the following command to start the application in watch mode. This ensures the backend and frontend are synchronized during development:
```bash
dotnet watch run --project S5Server/S5Server.csproj --non-interactive
```
*Note: If the project does not automatically rebuild after code changes, use the terminal restart command (Ctrl+R).*

### 4. Process Cleanup (MANDATORY)
To prevent orphan processes and port conflicts, you **MUST** terminate the application after each session, regardless of the execution result (success, error, or interruption).

Always execute one of the following commands before ending your terminal session:
```bash
pkill -f "dotnet run"
pkill -f "dotnet watch"
```
**This is a critical step. Do not consider a task complete until these commands have been executed.**
