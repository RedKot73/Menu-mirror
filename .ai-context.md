## Architecture Guidelines
The system is built as a highly coupled monolith, where the .NET Backend simultaneously serves a REST API, acts as a GraphQL Server, and hosts the compiled Angular Frontend.

- **"Thin" Backend Architecture**: Avoid complex architectural patterns like `CQRS (MediatR)` or `Repository Pattern`. Focus heavily on using `Entity Framework Core` directly in conjunction with `HotChocolate` via attributes (`[UseProjection]`, `[UseFiltering]`, `[UseSorting]`) to stream data effortlessly.
- **Business Logic Layer**: Complex queries or operations reside in `Services/` and should be implemented as **Extension Methods** layered on top of `DbSet<T>`.
- **Frontend SPA**: Powered by Angular 21.0.5. Source code lives in `/S5Server/Front/src/`. Upon building (via MSBuild Targets in the `.csproj`), the Angular app compiles straight into the backend's `wwwroot` directory to be statically served. **DO NOT** use Razor (`.cshtml`) or Blazor (`.razor`).
- **UI Framework Rules**: **Angular Material ONLY**. The use of Tailwind CSS, Bootstrap, or any custom utility-first framework is strictly forbidden unless explicitly mandated by the user. Rely on native Angular Material properties and components.
- **Database**: Relational data relies entirely on **PostgreSQL** configured via `Npgsql.EntityFrameworkCore.PostgreSQL`, with Code-First schema management via EF Core Migrations (`Migrations/`).

## Naming Conventions
To maintain uniformity across the monorepo, strictly adhere to the following rules:
- **C# (Backend)**: Use `PascalCase` for namespaces, classes, records, properties, methods, and interfaces (with `I` prefix).
- **TypeScript/Angular (Frontend)**: Use `camelCase` for variables, properties, and functions. Use `PascalCase` for Class and Interface names.
- **GraphQL API**: Use `camelCase` for all Queries and Mutations. 
  - **IMPORTANT**: HotChocolate automatically strips the `Get` prefix from C# methods (e.g., `GetTwoFactorSetup` becomes `twoFactorSetup` in the schema). Always verify names via introspection if a 400 error occurs.

## Security Constraints
- **CORS Policy**: Configured using `AllowedOrigins` from `appsettings.json`. Wildcard origin requests are blocked on production, allowing exceptions only for localized development servers (`localhost:4200`).
- **Authorization Schema**: Access uses JWT Bearer Tokens validated by `HotChocolate.AspNetCore.Authorization` middleware. APIs enforce HTTP `401/403` status outputs in lieu of redirecting automatically.
- **Secrets Management**: Local environment depends entirely on `UserSecretsId` or a local `.env` file (source of truth).
- **JWT Configuration**: Both `Issuer` and `Audience` are set to the same value (e.g., `S5Server`) because the backend is both the issuer and the consumer of tokens (monolithic context).

## Auth Mechanism (2FA & Security)

### 2-Step Login Sequence (GraphQL Mutation)
1. **Step 1: Credentials**: `AuthMutation.Login` returns an interim token if 2FA is enabled.
2. **Step 2: Verification**: `AuthMutation.VerifyTwoFactor` converts the interim token to a final JWT.

### Setup & Management (2FA)
- **Local QR Generation**: No external APIs (Google Charts removed). Uses `qrcode` npm library to render `otpauth://` URIs as Base64 DataURLs locally in the browser.
- **Time Drift Protection**: Server returns `ServerTimeIso`. Frontend calculates the drift and warns the user if it exceeds 30 seconds to prevent TOTP synchronization failures.
- **Key Persistence**: `GetTwoFactorSetup` (GQL: `twoFactorSetup`) returns existing keys to prevent invalidating user devices on re-entry.
- **Standardized Window**: Both setup and login use a `±30s` verification window for optimal balance of security and reliability.

### Route Guard Logic
| Guard | Condition | Result |
|-------|-----------|--------|
| `authGuard` | `requiresTwoFactor()` | → `/welcome` |
| `authGuard` | `isAuthenticated()` | Allow access |
| `authGuard` | Neither | → `/login` |
| `twoFactorGuard` | `requiresTwoFactor()` | Allow access to `/welcome` |
| `twoFactorGuard` | Not pending | → `/login` |

### Key Files
- `src/app/auth/auth.service.ts` — State signals: `user`, `pendingTwoFactor`, `requiresTwoFactor`, `isAuthenticated`.
- `src/app/auth/auth.guard.ts` — Global `authGuard` and route-specific `twoFactorGuard`.
- `src/app/pages/welcome/` — `WelcomeComponent` for TOTP entry.
- `S5Server/GraphQL/AuthMutation.cs` — `login` and `verifyTwoFactor` methods.
- `S5Server/Controllers/AccountController.cs` — Legacy REST endpoints (transitioning to GraphQL).

### Development Mode (Soft Mode)
- **Environment Variable**: `TWO_FACTOR_MODE=soft` in `.env`.
- **Behavior**: If 2FA check fails, a warning is logged: `[DEBUG] 2FA Check Failed for user {name}, but access granted due to Debug Mode.`. Access is granted regardless of code accuracy. **DO NOT USE IN PRODUCTION**.

## Infrastructure & DevOps
- **Kubernetes (K8s) & RDBMS**: PostgreSQL runs inside K8s, governed by **CloudNativePG (CNPG)**.
- **Containerization**: Database initialization uses `Dockerfile.kuber`.
- **CI/CD**: GitHub Actions automates builds (`.github/workflows/build-release.yml`), compiling self-contained binaries.
- **Telemetry & Logging**: Uses **Serilog** for logging and Azure Application Insights for production monitoring. Logs are stored in `Logs/s5server-.txt`.
- **Environment Configuration**: K8s bridges variables to `.NET` configuration (e.g., `PrimaryConnection__DB_Host`).

## Execution Policy & Development Workflow

This section strictly regulates the application lifecycle within various environments, specifically handling sandbox restrictions versus local development.

### 1. Environment Detection
Before executing any commands, identify the environment:
- **Restricted Sandbox**: Project path contains `/tmp/` or is running in `nsjail/docker`.
- **Local Dev**: Standard user path (e.g., `C:\`, `/home/user/`) or specified by the user.

### 2. Application Launch & Stop (MANDATORY)
To prevent port conflicts, environment misconfigurations, and avoid manual setup, **all Application Starts and Stops MUST be performed using the predefined helper scripts**. Do not manually export variables or type `dotnet watch run` in the terminal.

**Linux / Sandboxed Agent Environments**
- **Launch**: Run `bash ai/_run.s5app.sh {run|watch}`
    - `run`: Standard application start.
    - `watch`: Rebuilds frontend (`_rebuild_frontend.sh`) and starts with `dotnet watch`.
- **Stop**: Run `bash ai/_stop.s5app.sh`

**Windows / Developer Workstations (PowerShell для ОС Windows)**
- **Launch**: Run `.\ai\_run.s5app.ps1`
- **Stop**: Run `.\ai\_stop.s5app.ps1`

These scripts encapsulate all required environment variables (`/tmp` redirections, local NuGet sources, polling file watchers, and offline launch profiles).

### 3. Frontend Rebuild (MANDATORY after any Angular changes)

> [!IMPORTANT]
> **Any modification to Angular source files** (`.ts`, `.html`, `.scss` in `S5Server/Front/src/`) **requires a full frontend rebuild** before the changes take effect. `dotnet watch` does not always detect Angular changes reliably.

Run the provided rebuild script:
```bash
bash ai/_rebuild_frontend.sh
```

This script:
1. Stops all running dotnet/node processes (`_kill_dotnet.sh`)
2. Clears Angular cache (`.angular/`), `dist/`, and `wwwroot/`
3. Runs `ng build --configuration development` (with font inlining disabled for sandbox compatibility)
4. Copies the output to `S5Server/wwwroot/`

See [`ai/_rebuild_frontend.sh`](ai/_rebuild_frontend.sh) for the full implementation.

- **Eager Loading (Mandatory)**: Always ensure `MainDbContext` correctly loads navigation properties. For `TVezhaUser`, it is **MANDATORY** to load `Soldier` and its nested `Rank` before mapping to DTOs in authentication flows. Use `.Reference(u => u.Soldier).LoadAsync()` and `.Reference(s => s.Rank).LoadAsync()`.
- **Zero Hardcode Policy**: All configuration (JWT Secrets, Issuers, TOTP settings, timeouts) must be read from `IConfiguration`. Never use hardcoded strings for sensitive or environment-specific values.
- **GraphQL Debugging**: If receiving `400 Bad Request`, immediately verify:
  - **Naming Policy**: Did you include the `get` prefix? (e.g. use `twoFactorSetup`, NOT `getTwoFactorSetup` — HotChocolate strips the prefix).
  - **Payload Match**: Does the UI object match the C# `record` or `InputType` exactly?
  - **Authorization**: Is the JWT token present in the `Authorization` header?

### 5. Process Cleanup (MANDATORY)
To prevent port conflicts and orphan processes, **you MUST stop the application using the designated stop scripts from Section 2** before ending your interaction or attempting a fresh rebuild. DO NOT leave hanging `dotnet` processes.
**Do not consider a task complete until the stop script (`ai/_stop.s5app.sh` or `.ps1`) has been executed.**
